إصلاح v165 - ثبات المرفقات بعد تسجيل الخروج
===========================================

المشكلة التي تمت مراجعتها:
- بعض المرفقات كانت تختفي بعد تسجيل الخروج والدخول مرة أخرى.
- سبب الاختفاء قد يكون واحدا من أمرين:
  1) الملف لم يسجل في جدول attachments بعد رفعه إلى Storage.
  2) الملف موجود فعلا، لكن الواجهة لا تستطيع قراءة فهرسه أو إنشاء رابط العرض بسبب سياسات Supabase.

ما تم إصلاحه:
- إضافة فهرس دائم لبيانات المرفقات داخل حالة الموقع باسم attachmentRecordsV165.
- حفظ بيانات Storage path واسم الملف ونوعه وحجمه مع حالة الموقع السحابية.
- عند فشل قراءة جدول attachments، يحاول الموقع استرجاع بيانات المرفق من الفهرس الدائم بدلا من إظهار المرفق كأنه غير موجود.
- منع نافذة ملف الموظف من حذف معرفات المرفقات عند فشل القراءة المؤقت.
- توحيد تنزيل وفتح المرفقات عبر Storage signed URL أو public URL عند الحاجة.
- إضافة أداة فحص من Console:
  nawahAttachmentDiagnostics()

المرفقات المشمولة:
- مرفقات ملف الموظف: الهوية، الصورة، التوقيع، البصمة، العقد، الجوازات، الآيبان، وثائق الموظف، والإقرار.
- وثائق المنشأة.
- إعدادات المنشأة: الشعار، خلفية تسجيل الدخول، والعنوان الوطني.

خطوة Supabase المهمة:
- شغل الملف التالي مرة واحدة من Supabase SQL Editor:
  SUPABASE_ATTACHMENTS_STORAGE_SQL.sql

ماذا يفعل ملف SQL:
- ينشئ جدول public.attachments إذا لم يكن موجودا.
- ينشئ buckets:
  employee-attachments
  company-documents
- يضيف سياسات قراءة ورفع للمستخدمين المسجلين.
- يجعل تعديل وحذف المرفقات للمدير فقط عند وجود جدول app_user_profiles.

طريقة التحقق:
1) سجل الدخول.
2) ارفع مرفقا في ملف موظف أو وثائق المنشأة أو إعدادات المنشأة.
3) سجل الخروج ثم ادخل مرة أخرى.
4) افتح نفس الصفحة وتأكد من ظهور المرفق.
5) عند الحاجة افتح Console وشغل:
   nawahAttachmentDiagnostics()
   إذا كانت found = true و hasUrl = true فالمرفق محفوظ ويمكن عرضه.
